# 液体散装化学品船运费计算系统 - 项目审核文档

**项目名称**：液体散装化学品船运费计算系统（Web 端）  
**开发周期**：2026 年 3 月  
**技术栈**：React 19 + TypeScript + Express 4 + tRPC 11 + MySQL 8.0  
**部署方式**：云服务器（支持 Vercel、Railway 等）  
**项目状态**：MVP 版本完成，48 项单元测试全部通过

---

## 1. 项目概述

### 1.1 业务背景

液体散装化学品运输涉及复杂的运费计算逻辑。不同船东对同一航线的费率不同，且费率随装货量呈阶梯式变化。现有系统缺乏标准化的费率管理和计算工具，导致报价效率低下、易出错。本系统通过数字化费率表管理和自动化运费计算，提升报价准确性和效率。

### 1.2 核心目标

- **费率管理**：支持多船东、多航线的费率表维护，包含动态阶梯定价
- **运费计算**：基于线性内插法的精确费率计算，处理边界条件
- **权限管理**：区分普通操作员和系统管理员的功能权限
- **数据追溯**：完整的计算历史记录和审计日志
- **易用性**：直观的用户界面，支持批量导入和导出

### 1.3 用户群体

| 用户角色 | 主要职责 | 可用功能 |
|---------|---------|---------|
| 普通操作员 | 报价、运费查询 | 运费计算、历史查询 |
| 系统管理员 | 数据维护、权限管理 | 船东/港口/费率表管理、账户管理、批量导入 |

### 3.1 已实现功能（MVP）

**港口管理**（仅管理员）

- 支持港口代码、中英文名称、所属国家

**费率表管理**（仅管理员）
- 创建、编辑、删除费率表

- 边界处理：低于最低节点、高于最高节点的特殊处理

#### 3.1.4 计算历史与审计

## 4. 数据库设计

### 4.1 核心数据表

#### 4.1.1 用户表 (users)

```sql
CREATE TABLE users (
  id INT AUTO_INCREMENT PRIMARY KEY,
  openId VARCHAR(64) UNIQUE NOT NULL,
  name TEXT,
  username VARCHAR(64),
  email VARCHAR(320),
  phone VARCHAR(20),
  password_hash VARCHAR(255),
  loginMethod VARCHAR(64),
  role ENUM('user', 'admin') DEFAULT 'user',
  is_active TINYINT DEFAULT 1,
  createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  lastSignedIn TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**关键字段说明**：
- `openId`：Manus OAuth 唯一标识
- `username`、`phone`、`password_hash`：本地账户登录凭证
- `role`：用户角色（user/admin）
- `is_active`：账户启用状态（1=启用，0=禁用）

#### 4.1.2 船东表 (ship_owners)

```sql
CREATE TABLE ship_owners (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  code VARCHAR(50) UNIQUE NOT NULL,
  contact VARCHAR(50),
  phone VARCHAR(20),
  remark VARCHAR(255),
  is_deleted TINYINT DEFAULT 0,
  createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

#### 4.1.3 港口表 (ports)

```sql
CREATE TABLE ports (
  id INT AUTO_INCREMENT PRIMARY KEY,
  code VARCHAR(20) UNIQUE NOT NULL,
  name_cn VARCHAR(100) NOT NULL,
  name_en VARCHAR(100),
  country VARCHAR(50),
  is_deleted TINYINT DEFAULT 0,
  createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

#### 4.1.4 费率表主表 (rate_tables)

```sql
CREATE TABLE rate_tables (
  id INT AUTO_INCREMENT PRIMARY KEY,
  owner_id INT NOT NULL,
  load_port_id INT NOT NULL,
  discharge_port_id INT NOT NULL,
  valid_from DATE NOT NULL,
  valid_to DATE NOT NULL,
  status TINYINT DEFAULT 1,
  is_deleted TINYINT DEFAULT 0,
  created_by INT,
  createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_by INT,
  updatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY idx_match_route (owner_id, load_port_id, discharge_port_id, status, valid_from, valid_to)
);
```

**关键设计**：
- 联合唯一索引 `idx_match_route` 支持快速费率表匹配查询
- `valid_from` 和 `valid_to` 支持时间范围管理
- `status` 字段支持费率表启用/禁用

#### 4.1.5 费率阶梯表 (rate_steps)

```sql
CREATE TABLE rate_steps (
  id INT AUTO_INCREMENT PRIMARY KEY,
  rate_table_id INT NOT NULL,
  quantity DECIMAL(10, 2) NOT NULL,
  rate DECIMAL(10, 4) NOT NULL,
  createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (rate_table_id) REFERENCES rate_tables(id) ON DELETE CASCADE
);
```

**关键设计**：
- `quantity`：装货量阶梯（精度 0.01 MT）
- `rate`：对应费率（精度 0.0001 元/MT）
- 级联删除：删除费率表时自动删除所有阶梯行

#### 4.1.6 计算历史表 (calculation_history)

```sql
CREATE TABLE calculation_history (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT,
  owner_id INT NOT NULL,
  load_port_id INT NOT NULL,
  discharge_port_id INT NOT NULL,
  confirmed_quantity VARCHAR(50) NOT NULL,
  calculated_rate VARCHAR(50) NOT NULL,
  total_freight VARCHAR(50) NOT NULL,
  calculation_details JSON,
  createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id),
  FOREIGN KEY (owner_id) REFERENCES ship_owners(id),
  FOREIGN KEY (load_port_id) REFERENCES ports(id),
  FOREIGN KEY (discharge_port_id) REFERENCES ports(id)
);
```

**关键设计**：
- 所有数值字段使用 VARCHAR 存储以保持精度
- `calculation_details` 存储 JSON 格式的计算过程
- 完整的审计追踪（用户、时间、计算参数）

### 4.2 数据库索引策略

| 表名 | 索引名 | 索引字段 | 用途 |
|-----|--------|---------|------|
| rate_tables | idx_match_route | (owner_id, load_port_id, discharge_port_id, status, valid_from, valid_to) | 运费计算时快速匹配费率表 |
| rate_steps | idx_rate_table | (rate_table_id) | 快速查询费率表的所有阶梯 |
| calculation_history | idx_user_created | (user_id, createdAt) | 用户历史查询 |
| calculation_history | idx_owner_created | (owner_id, createdAt) | 船东历史查询 |

## 6. 核心算法：线性内插法

### 6.1 算法原理

线性内插法用于在两个已知数据点之间估算未知值。在运费计算中，当确认装货量介于两个阶梯节点之间时，使用该方法计算费率。

**公式**：

$$r = r_1 + \frac{(q - q_1) \times (r_2 - r_1)}{q_2 - q_1}$$

其中：
- $q$：确认装货量
- $q_1, q_2$：下限和上限节点的装货量
- $r_1, r_2$：下限和上限节点的费率
- $r$：计算得到的费率

### 6.2 边界条件处理

| 情况 | 处理方式 | 示例 |
|-----|---------|------|
| 精确匹配 | 直接返回节点费率 | 装货量 3000 MT，存在节点 3000 → 费率 225 |
| 低于最低节点 | 返回最低节点费率 | 装货量 2000 MT，最低节点 3000 → 费率 225 |
| 高于最高节点 | 返回最高节点费率 | 装货量 12000 MT，最高节点 11000 → 费率 166 |
| 内插 | 使用线性内插公式 | 装货量 3500 MT，节点 3000(225) 和 4000(210) → 费率 217.5 |

### 6.3 实现细节

- **精度**：费率计算精度为小数点后 4 位（0.0001 元/MT）
- **舍入**：使用标准四舍五入规则
- **总运费**：总运费 = 确认装货量 × 计算费率

### 6.4 测试覆盖

系统包含 10 项单元测试覆盖内插法算法：

- ✅ 精确匹配场景
- ✅ 线性内插场景
- ✅ 低于最低节点
- ✅ 高于最高节点
- ✅ 单个节点的特殊处理
- ✅ 浮点数精度验证
- ✅ 异常输入处理

## 7. 安全性设计

### 7.1 认证与授权

- **OAuth 集成**：支持 Manus OAuth 第三方登录，安全可靠
- **本地账户**：密码使用 bcryptjs 加密存储（盐值 10 轮）
- **会话管理**：基于 JWT token 的无状态会话
- **权限控制**：后端通过 `adminProcedure` 和 `protectedProcedure` 强制权限校验
- **账户禁用**：禁用账户在认证层被拦截，无法访问系统

### 7.2 数据保护

- **SQL 注入防护**：使用 Drizzle ORM 的参数化查询
- **XSS 防护**：React 自动转义用户输入
- **CSRF 防护**：tRPC 框架内置 CSRF 保护
- **数据加密**：密码使用 bcryptjs 加密，敏感数据在传输层使用 HTTPS
- **软删除**：关键数据（船东、港口）采用软删除，保留历史数据

### 7.3 审计与日志

- **操作日志**：所有修改操作记录操作人、时间、内容
- **计算历史**：完整的运费计算审计追踪
- **用户追踪**：每个用户的登录、操作记录可追溯

### 7.4 环境隔离

- **环境变量**：敏感信息（数据库密码、API 密钥）通过环境变量管理
- **开发/生产分离**：不同环境使用不同的数据库和配置
- **错误处理**：生产环境不暴露详细的技术错误信息

## 8. 性能优化

### 8.1 数据库优化

- **联合索引**：`rate_tables` 表的联合索引支持 O(log n) 的费率表匹配查询
- **查询优化**：使用 JOIN 而非 N+1 查询，减少数据库往返
- **分页查询**：历史记录支持分页，避免一次性加载大量数据

### 8.2 前端优化

- **代码分割**：使用动态导入实现路由级代码分割
- **缓存策略**：利用 React Query 的缓存机制减少重复请求
- **虚拟滚动**：长列表使用虚拟滚动提升性能
- **图片优化**：使用 WebP 格式和懒加载

### 8.3 后端优化

- **连接池**：MySQL 连接池管理，提升并发处理能力
- **缓存**：关键数据（港口列表、船东列表）可缓存
- **异步处理**：文件导入导出使用异步处理，避免阻塞主线程

## 附录 B：常见问题

**Q1：如何修改运费计算的精度？**  
A：修改 `server/routers.ts` 中的 `interpolateRate` 函数，调整 `toFixed()` 参数。

**Q2：如何支持多币种？**  
A：在 `rate_tables` 表中添加 `currency` 字段，前端展示时进行汇率转换。

**Q3：如何处理大量历史数据？**  
A：使用分页查询，或将历史数据归档到专用表中。
